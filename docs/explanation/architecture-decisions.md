# 架构设计决策

本文档深入解释 ADN 项目中的关键架构决策，包括设计动机、权衡分析和替代方案。

---

## 1. 为什么使用块注意力残差 (AttnRes)？

### 问题背景

标准 Transformer 的自注意力机制存在以下问题：

1. **内存复杂度 O(L²)**: 长序列时内存爆炸
2. **计算冗余**: 远距离 token 的注意力权重通常很小
3. **注意力稀释**: 序列越长，单个 token 获得的注意力越少

### 解决方案对比

| 方案 | 复杂度 | 优点 | 缺点 |
|------|--------|------|------|
| 稀疏注意力 | O(L√L) | 保持全局连接 | 模式固定，不够灵活 |
| 线性注意力 | O(L) | 计算高效 | 表达能力下降 |
| **AttnRes** | O(L×N) | 平衡效率和效果 | 需要调优块数 |
| 滑动窗口 | O(L×W) | 简单高效 | 丢失长距离依赖 |

### AttnRes 的核心洞察

```
关键观察: 注意力矩阵通常是低秩的

标准注意力: 每个查询关注所有 L 个键
AttnRes:     每个查询先关注 N 个块，再关注块内 token

当 N << L 时，显著降低计算量，同时保持表达能力
```

### 设计决策

**选择 N=8 的原因**:
- 论文实验显示 N=8 能恢复 95%+ 的 FullAttnRes 性能
- 在效率和效果之间达到最佳平衡点
- 硬件友好 (2 的幂次)

## 2. 为什么冻结 KV 缓存？

### qTTT 的设计哲学

查询时训练 (qTTT) 的核心假设：

```
假设: 预训练已经学到了良好的键和值表示
      只需要针对特定查询调整查询表示
```

### 冻结 vs 不冻结

| 策略 | 优点 | 缺点 |
|------|------|------|
| **冻结 KV** | 计算成本低、缓存一致、避免灾难性遗忘 | 适应范围有限 |
| 更新 KV | 更强的适应能力 | 破坏缓存、计算成本高、不稳定 |
| 部分更新 | 折中方案 | 需要复杂的更新策略 |

### 边际最大化损失

为什么选择边际损失而不是标准交叉熵？

```python
# 交叉熵: 只关注正确标签
loss = -log(p_correct)

# 边际损失: 显式拉大正确与错误的差距
margin = logit_correct - max(logit_incorrect)
loss = max(0, margin_target - margin)
```

**原因**: 长上下文中需要更大的决策边界来对抗注意力稀释。

## 3. 为什么使用 RaBitQ 压缩？

### KV 缓存压缩方案对比

| 方案 | 压缩率 | 重建质量 | 计算开销 |
|------|--------|----------|----------|
| 8-bit 量化 | 4x | 高 | 低 |
| 4-bit 量化 | 8x | 中高 | 低 |
| **RaBitQ (1-bit)** | ~32x | 中 | 中 |
| 向量量化 | 可变 | 中 | 高 |
| 稀疏化 | 可变 | 高 | 中 |

### RaBitQ 的核心创新

1. **随机旋转**: 使各维度方差均匀化
2. **二进制量化**: 极致压缩
3. **残差补偿**: 保留关键信息

```
压缩流程:
x ──> 随机旋转 ──> 符号量化 ──> 1-bit 表示
│                              ↓
└──── 残差存储 <────────────── 重建
```

### 配置选择

- **k1 (1-bit)**: 边缘设备，极致压缩
- **k2 (2-bit)**: 平衡选择，推荐默认
- **k3 (3-bit)**: 高精度场景

## 4. 为什么使用重构损失作为门控信号？

### 动态计算的挑战

问题：如何知道一个 token 需要更多计算？

### 信号选择对比

| 信号 | 优点 | 缺点 |
|------|------|------|
| **重构损失** | 自包含、无需标签、反映模型不确定性 | 计算成本 |
| 熵 | 简单 | 可能误判 |
| 最大概率 | 简单 | 不够鲁棒 |
| 外部评估器 | 准确 | 需要额外训练 |

### 重构损失的数学解释

```
重构损失 = KL(模型输出 || 模型输出的平滑版本)

高损失 = 模型输出尖锐且不稳定
       = 模型处于"困惑"状态
       = 需要额外计算
```

## 5. 架构演进历史

### 版本 0.1: 基础 AttnRes

- 仅包含块注意力残差
- 验证 AttnRes 的有效性

### 版本 0.2: 添加 qTTT

- 引入查询时训练
- 解决长上下文适应问题

### 版本 0.3: 添加 RaBitQ

- 引入 KV 缓存压缩
- 支持更长上下文

### 版本 0.4: 添加 Engram

- 引入 n-gram 记忆
- 增强重复模式处理能力

### 版本 0.5: 添加 Gating

- 引入动态计算门控
- 实现自适应计算分配

### 当前版本: 统一架构

- 所有模块协同工作
- 统一配置和 API

## 6. 未来方向

### 可能的改进

1. **自适应块数**: 根据输入动态调整 AttnRes 块数
2. **分层 qTTT**: 不同层使用不同的 qTTT 配置
3. **学习压缩**: 端到端学习最优压缩策略
4. **多模态扩展**: 将 ADN 扩展到视觉和音频

- 如何量化每个模块的贡献？
